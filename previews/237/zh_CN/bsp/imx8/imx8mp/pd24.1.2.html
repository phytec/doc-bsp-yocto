本BSP手册指导您完成板级支持包（BSP）的安装和创建步骤，并描述如何使用 phyCORE-i.MX8M Plus Kit 的接口。此外，本文件还描述了如何从源代码创建BSP镜像。这对于需要更改默认镜像并需要以简单和可重复的方式实现这些更改的用户非常有用。此外，本手册的某些部分需要在个人计算机（主机）上执行命令。所有这些命令都假定在Linux操作系统上执行。

备注

本文件包含代码示例，描述如何通过串行终端与板子进行通信。代码示例的行以“host:~$”、“target:~$”或“u-boot=>”开头。这些关键字描述了命令执行的位置。只有在这些关键字之后，才需要复制实际的命令。

2. 开始使用

该 phyCORE-i.MX8M Plus Kit 随附预先刷写的SD卡。它包含 phytec-qt6demo-image ，可以直接用作启动源。默认情况下，eMMC仅编程了U-Boot。您可以从 PHYTEC下载服务器 获取所有源代码。本章将解释如何将BSP镜像刷写到SD卡以及如何启动开发板。

有几种方法可以将镜像写入SD卡或甚至eMMC。最显著的是使用Linux命令行工具 dd 进行简单的顺序写入。另一种方法是使用PHYTEC的系统初始化程序 partup ，这使得格式化更复杂的系统变得特别简单。您可以从其发布页面获取 partup的预构建Linux二进制文件 。还请阅读 partup的自述文件 以获取安装说明。

2.1. 下载镜像

该镜像包含所有必要的文件，并确保分区和任何原始数据被正确写入。可以从 PHYTEC 下载服务器 下载 partup 包和可以使用 dd 刷写的 WIC 镜像。

从下载服务器获取partup包或WIC镜像：

host:~$ wget https://download.phytec.de/Software/Linux/BSP-Yocto-i.MX8MP/BSP-Yocto-Ampliphy-i.MX8MP-PD24.1.2/images/ampliphy-xwayland/phyboard-pollux-imx8mp-3/phytec-qt6demo-image-phyboard-pollux-imx8mp-3.rootfs.partup
host:~$ wget https://download.phytec.de/Software/Linux/BSP-Yocto-i.MX8MP/BSP-Yocto-Ampliphy-i.MX8MP-PD24.1.2/images/ampliphy-xwayland/phyboard-pollux-imx8mp-3/phytec-qt6demo-image-phyboard-pollux-imx8mp-3.rootfs.wic.xz

备注

对于eMMC，更复杂的分区方案或甚至只是大型镜像，我们建议使用partup软件包，因为它在写入速度上比 dd 更快，并且允许对闪存设备进行更灵活的配置。

2.2. 将镜像写入SD卡

警告

要创建可启动的SD卡，您必须在Linux主机PC上拥有root权限。在指定目标设备时请务必小心！所选设备上的所有文件将立即被删除，而不会有任何进一步的确认！

选择错误的设备可能会导致 数据丢失 ，例如，可能会擦除您当前在主机PC上运行的系统！

2.2.1. 寻找正确的设备

要创建可启动的SD卡，您首先必须找到SD卡及其可能的分区的正确设备名称。如果SD卡的任何分区已挂载，请在开始将镜像复制到SD卡之前卸载这些分区。

  1. 为了获取正确的设备名称，请移除您的SD卡并执行：

    host:~$ lsblk

  2. 现在插入你的SD卡，然后再次执行命令：

    host:~$ lsblk

  3. 比较两个输出，以查找第二个输出中列出的新设备名称。这些是SD卡的设备名称（如果SD卡已格式化，则包括设备和分区）。

  4. 为了验证找到的设备名称，请执行命令 sudo dmesg。在其输出的最后几行中，您应该也能找到设备名称，例如 /dev/sde 或 /dev/mmcblk0 （具体取决于您的系统）。

或者，您可以使用您选择的图形程序，例如 GNOME Disks 或 KDE Partition Manager 来找到正确的设备。

现在您已经获得了正确的设备名称，例如 /dev/sde，您可以查看必须在格式化SD卡时卸载的分区。在这种情况下，您还会在输出中找到带有附加数字的设备名称（例如 /dev/sde1）。这些代表分区。一些Linux发行版在设备插入时会自动挂载分区。然而，在写入之前，必须卸载这些分区，以避免数据损坏。

卸载所有这些分区，例如：

host:~$ sudo umount /dev/sde1
host:~$ sudo umount /dev/sde2

现在，SD卡已经准备好可以使用 partup 、 dd 或 bmap-tools 来写入镜像。

2.2.2. 使用bmap-tools

准备SD卡的一种方法是使用 bmap-tools 。Yocto会自动为WIC镜像创建一个块映射文件（ <IMAGENAME>-<MACHINE>.wic.bmap ），该文件描述了镜像内容并包含数据完整性的校验和。 bmaptool 已被多种Linux发行版打包。对于基于Debian的系统，可以通过以下命令安装：

host:~$ sudo apt install bmap-tools

通过以下命令将WIC镜像烧录到SD卡：

host:~$ bmaptool copy phytec-qt6demo-image-phyboard-pollux-imx8mp-3?(.rootfs).wic?(.xz) /dev/<your_device>

将 <your_device>替换为您之前找到的实际SD卡设备名称，并确保将文件 <IMAGENAME>-<MACHINE>.wic.bmap 与常规WIC镜像文件放在一起，以便bmaptool知道哪些块需要写入，哪些块需要跳过。

警告

bmaptool 仅覆盖SD卡上镜像数据所在的区域。这意味着在写入镜像后，之前写入的U-Boot环境可能仍然可用。

2.2.3. 使用partup

使用partup写入SD卡只需一个命令：

host:~$ sudo partup install phytec-qt6demo-image-phyboard-pollux-imx8mp-3?(.rootfs).partup /dev/<your_device>

确保将 <your_device>替换为您之前找到的实际设备名称。

关于partup的进一步使用说明，请参阅其 官方文档网站 。

警告

使用resize2fs版本1.46.6及更早版本的主机系统（例如Ubuntu 22.04）无法将使用Yocto Mickledore或更新版本创建的partup软件包写入SD卡。这是由于resize2fs中的一个新默认选项导致的不兼容性。有关详细信息，请参阅 发行说明 。

备注

partup 的优点在于可以清除MMC用户区中的特定原始区域，这在我们提供的partup软件包中用于擦除任何现有的U-Boot环境。这是一个已知的问题， bmaptool 并不能解决，如前一章所提到的。

partup相较于其他刷写工具的另一个主要优势是，它允许配置特定于MMC的部分，比如在写入时可以直接写入eMMC引导分区，而无需调用多个其他命令。

2.2.4. 使用 dd

在卸载所有SD卡的分区后，您可以创建可启动的SD卡。

一些PHYTEC BSP生成未压缩的镜像（文件名扩展名为*.wic），而另一些则生成压缩的镜像（文件名扩展名为*.wic.xz）。

要写入未压缩的镜像（*.wic），请使用以下命令：

host:~$ sudo dd if=phytec-qt6demo-image-phyboard-pollux-imx8mp-3?(.rootfs).wic of=/dev/<your_device> bs=1M conv=fsync status=progress

或者要写入压缩后的镜像（*.wic.xz），请使用以下命令：

host:~$ xzcat phytec-qt6demo-image-phyboard-pollux-imx8mp-3?(.rootfs).wic.xz | sudo dd of=/dev/<your_device> bs=1M conv=fsync status=progress

再次确保将<your_device>替换为之前找到的实际设备名称。

参数 conv=fsync 强制在 dd 返回之前对设备进行同步操作。这确保所有数据块都已写入SD卡，而没有任何数据留在内存中。参数 status=progress 将打印出有关已复制和仍需复制的数据量的信息，直到操作完成。

3. 构建BSP（板级支持包）

本节将指导您通过使用Yocto和phyLinux脚本进行 i.MX 8M Plus BSP的一般构建过程。如需有关我们的meta-layer或Yocto的一般信息，请访问： Yocto Reference Manual (scarthgap) 。

3.1. 基本设置

如果您从未在您的计算机上使用Yocto创建过Phytec BSP，您应该仔细查看 Yocto Reference Manual (scarthgap) 中的BSP Workspace安装章节。

3.2. 下载BSP

获取BSP源代码有两种方式。您可以从我们的下载页面下载完整的BSP源代码： BSP-Yocto-IMX8MP ；或者您可以使用Yocto自行获取并构建。这在您想要进行自定义时特别有用。

phyLinux脚本是一个用于PHYTEC Yocto BSP的基本管理工具，使用Python编写。它主要是一个辅助工具，用于初始化BSP。

  • 创建一个新的项目文件夹，获取phyLinux，并使脚本可执行：

    host:~$ mkdir ~/yocto
host:~$ cd yocto/
host:~/yocto$ wget https://download.phytec.de/Software/Linux/Yocto/Tools/phyLinux
host:~/yocto$ chmod +x phyLinux

    警告

    一个干净的文件夹很重要，因为phyLinux会清理其工作目录。从一个不为空的目录调用phyLinux将会导致警告。

  • 运行phyLinux：

    host:~/yocto$ ./phyLinux init

    备注

    在首次初始化时，phyLinux脚本会要求您在 /usr/local/bin 目录中安装Repo工具。

  • 在执行初始化命令时，您需要选择您的处理器平台（SoC）、PHYTEC的BSP版本号以及您正在使用的硬件。

    备注

    如果您无法根据菜单中提供的信息识别您的开发板，请查看产品的发票。并查看 our BSP 。

  • 也可以通过命令行参数直接传递这些信息：

    host:~/yocto$ DISTRO=ampliphy-xwayland MACHINE=phyboard-pollux-imx8mp-3 ./phyLinux init -p imx8mp -r BSP-Yocto-Ampliphy-i.MX8MP-PD24.1.2

在执行init命令后，phyLinux将打印一些重要的说明。例如，它将打印您的git身份、选择的SOC和BSP版本，以及构建过程下一步的信息。

3.2.1. 开始构建过程

  • 设置Shell环境变量：

    host:~/yocto$ source sources/poky/oe-init-build-env

    备注

    在每次打开新的用于编译的终端时，都需要执行。

  • 当前的工作目录应该更改为 build/。

  • 打开主配置文件，接受GPU和VPU二进制许可证协议。通过取消注释相应的行来完成此操作，如下所示。

    host:~/yocto/build$ vim conf/local.conf
# Uncomment to accept NXP EULA
# EULA can be found under ../sources/meta-freescale/EULA
ACCEPT_FSL_EULA = "1"

  • 构建您的镜像：

    host:~/yocto/build$ bitbake phytec-qt6demo-image

    备注

    对于第一次构建，我们建议从我们的较小的非图形化镜像phytec-headless-image开始，以查看一切是否正常工作。

    host:~/yocto/build$ bitbake phytec-headless-image

    第一次编译过程在现代的Intel Core i7处理器上大约需要40分钟。所有后续的构建将使用缓存，应该大约需要3分钟。

4.2. 烧写eMMC

要从 eMMC 启动，请确保 BSP 镜像已正确烧写到 eMMC，并且 bootmode switch (S3) 设置为 eMMC。

警告

当eMMC和SD卡上烧录了相同（完全相同）的镜像时，启动分区的UUID也是相同的。如果在启动时连接了SD卡，这会导致非确定性的行为，因为Linux会根据UUID挂载启动分区。

target:~$ blkid

可以运行以检查当前设置是否受到影响。如果 mmcblk2p1 和 mmcblk1p1 具有相同的UUID，则该设置受到影响。

4.2.1. 从网络烧写 eMMC

i.MX 8M Plus 板具有以太网连接器，可以通过网络进行更新。确保正确设置开发主机。IP需要设置为192.168.3.10，子网掩码为255.255.255.0，并且需要有一个TFTP服务器。从高层次来看，eMMC设备就像SD卡。因此，可以直接将 WIC镜像 （ <name>.wic ）从Yocto构建系统烧写到eMMC。该镜像包含bootloader、内核、设备树、设备树覆盖和根文件系统。

5. 开发

5.1. 主机网络准备

为了在bootloader中执行涉及网络的各种任务，需要设置一些主机服务。在开发主机上，必须安装和配置TFTP、NFS和DHCP服务器。启动以太网所需的工具如下：

host:~$ sudo apt install tftpd-hpa nfs-kernel-server kea

5.1.1. TFTP服务器设置

  • 首先，创建一个目录来存储TFTP文件：

    host:~$ sudo mkdir /srv/tftp

  • 然后将您的BSP镜像文件复制到此目录，并确保其他用户对tftp目录中的所有文件具有读取权限，否则它们将无法从目标访问。

    host:~$ sudo chmod -R o+r /srv/tftp

  • 您还需要为相应的接口配置一个静态IP地址。PHYTEC评估板的默认IP地址是192.168.3.11。设置主机地址为192.168.3.10，子网掩码为255.255.255.0是一个不错的选择。

    host:~$ ip addr show <network-interface>

    将<network-interface>替换为您配置并希望连接到开发板的网络接口。您可以通过不指定网络接口来显示所有网络接口。

  • 您收到的消息应包含以下内容：

    inet 192.168.3.10/24 brd 192.168.3.255

  • 创建或编辑 /etc/default/tftpd-hpa 文件：

    # /etc/default/tftpd-hpa

TFTP_USERNAME="tftp"
TFTP_DIRECTORY="/srv/tftp"
TFTP_ADDRESS=":69"
TFTP_OPTIONS="-s -c"

  • 将 TFTP_DIRECTORY 设置为您的 TFTP 服务器根目录

  • 将TFTP_ADDRESS设置为服务器正在监听的主机地址（设置为0.0.0.0:69以监听所有本地IP）。

  • 设置 TFTP_OPTIONS，以下命令显示可用的选项：

    host:~$ man tftpd

  • 重新启动服务以应用配置更改：

    host:~$ sudo service tftpd-hpa restart

现在将开发板的以太网端口连接到您的主机系统。我们还需要在嵌入式板和TFTP服务器之间建立网络连接。服务器的IP地址应设置为192.168.3.10，子网掩码为255.255.255.0。

5.1.1.1. NFS服务器设置

  • 创建一个NFS目录：

    host:~$ sudo mkdir /srv/nfs

  • NFS服务器并不限制于某个特定的文件系统位置，因此在大多数发行版中，我们只需修改文件 /etc/exports ，并将我们的根文件系统导出到嵌入式网络。在这个示例文件中，整个目录被导出，而开发主机的“实验室网络”地址为192.168.3.10。IP地址需要根据本地需求进行调整：

    /srv/nfs 192.168.3.0/255.255.255.0(rw,no_root_squash,sync,no_subtree_check)

  • 现在NFS服务器需要再次读取 /etc/exportfs 文件：

    host:~$ sudo exportfs -ra

5.1.1.2. DHCP服务器设置

  • 创建或编辑 /etc/kea/kea-dhcp4.conf 文件；使用内部子网示例。将<network-interface>替换为物理网络接口的名称：

    {
  "Dhcp4": {
    "interfaces-config": {
      "interfaces": [ "<network-interface>/192.168.3.10" ]
    },
    "lease-database": {
      "type": "memfile",
      "persist": true,
      "name": "/tmp/dhcp4.leases"
    },
    "valid-lifetime": 28800,
    "subnet4": [{
        "id": 1,
        "next-server": "192.168.3.10",
        "subnet": "192.168.3.0/24",
        "pools": [
          { "pool": "192.168.3.1 - 192.168.3.255" }
        ]
    }]
  }
}

警告

在创建子网时请小心，因为这可能会干扰公司网络政策。为了安全起见，请使用不同的网络，并通过 interfaces 配置选项指定该网络。

  • 现在DHCP服务器需要重新读取 /etc/kea/kea-dhcp4.conf 文件：

    host:~$ sudo systemctl restart kea-dhcp4-server

当您启动/重启主机PC时，如果没有按照kea-dhcp4配置中指定的网络接口处于活动状态，kea-dhcp4-server将无法启动。请确保在连接接口时启动/重启systemd服务。

5.3.5. 通过UUU工具将U-boot镜像烧写到eMMC

警告

UUU将U-boot刷入eMMC BOOT（硬件）启动分区后，会在eMMC中设置BOOT_PARTITION_ENABLE。这带来一个问题，因为我们希望bootloader驻留在eMMC USER分区中。如果刷入下一个U-Boot版本的.wic镜像而不禁用BOOT_PARTITION_ENABLE位，将导致设备始终使用保存在BOOT分区中的U-boot。要在U-Boot中解决此问题：

u-boot=> mmc partconf 2 0 0 0
u-boot=> mmc partconf 2
EXT_CSD[179], PARTITION_CONFIG:
BOOT_ACK: 0x0
BOOT_PARTITION_ENABLE: 0x0
PARTITION_ACCESS: 0x0

或从Linux检查 Disable booting from eMMC boot partitions。

这样bootloader仍然会被刷写到 eMMC 启动分区，但不会被使用！

在使用 partup 工具和 .partup 包进行eMMC烧写时，这一过程是默认进行的，这使得partup再次成为优越的烧写选项。

host:~$ sudo uuu -b emmc imx-boot

6. 设备树 (DT)

6.1. 介绍

以下文本简要描述了设备树，关于设备树的相关文档可以在Linux内核文档中找到（https://docs.kernel.org/devicetree/usage-model.html）。

“Open Firmware Device Tree”或简称设备树（DT）是一种用于描述硬件的数据结构和语言。更具体地说，它是一个可由操作系统读取的硬件描述，以便操作系统不需要对机器的细节进行硬编码

内核文档是设备树介绍的一个非常好的资源。关于设备树数据格式的概述可以在 devicetree.org 的设备树使用页面找到。

6.2. PHYTEC i.MX 8M Plus BSP设备树概念

以下部分解释了PHYTEC在如何为我们的基于SoC i.MX 8M Plus 的核心板设备树所定义的一些规则。

6.2.1. 设备树结构

  • Module.dtsi - 模块包括所有安装在核心板上的设备，例如PMIC和RAM。

  • Board.dts - 包含核心板 dtsi 文件。底板使用的来自SoC i.MX 8M Plus 的设备也包含在此 dts 中。

  • Overlay.dtso - 根据安装或未连接在核心板或底板上的可选硬件（例如 SPI 闪存或 PEB-AV-10）的情况，启用/禁用功能。

从Linux内核的根目录，我们的 i.MX 8 平台的设备树文件可以在 arch/arm64/boot/dts/freescale/ 找到。

6.2.2. 设备树Overlay

Device Tree overlays are device tree fragments that can be merged into a device tree during boot time. These are for example hardware descriptions of an expansion board. They are instead of being added to the device tree as an extra include, now applied as an overlay. They also may only contain setting the status of a node depending on if it is mounted or not. The device tree overlays are placed next to the other device tree files in our Linux kernel repository in the folder arch/arm64/boot/dts/freescale/.

为了实现最大的软件重用，Linux内核提供了一个巧妙的软件架构，将软件组件分层为特定于板的部分。BSP（板级支持包）尽可能地对套件的功能进行模块化。当开发定制的主板或甚至客户特定的模块时，大部分软件支持可以在不易出错的复制粘贴情况下重新使用。与板相关的内核代码可以在内核代码库中的设备树（DT）中找到，路径为 arch/arm64/boot/dts/freescale/*.dts 。

实际上，软件重用是Linux内核最重要的特性之一，尤其是在ARM实现中，它必须应对大量复杂的系统级芯片（SoC）CPU的可能性。整个特定于板的硬件在设备树（DT）中描述，并不属于内核镜像本身。硬件描述在一个单独的二进制文件中，称为设备树二进制文件（Device Tree Blob，DTB）（参见 device tree）。

7.1. i.MX 8M Plus 引脚复用

该 i.MX 8M Plus 系统芯片包含许多外设接口。为了在保持最大功能的同时减少封装尺寸和降低整体系统成本，许多 i.MX 8M Plus 引脚可以多路复用多达八种信号功能。尽管存在许多可能的引脚多路复用组合，但由于时序限制，只有一定数量的组合被称为有效的 IO 集合。这些有效的 IO 集合经过精心选择，以为用户提供多种可能的应用场景。

请参考我们的硬件手册或NXP i.MX 8M Plus 参考手册，以获取有关特定引脚和复用能力的更多信息。

IO设置配置，也称为复用（muxing），是在设备树中完成的。驱动程序pinctrl-single读取设备树的节点fsl,pins，并进行适当的引脚复用。

以下是 imx8mp-phyboard-pollux-rdk.dts中UART1设备的引脚复用示例：

pinctrl_uart1: uart1grp {
        fsl,pins = <
                MX8MP_IOMUXC_UART1_RXD_UART1_DCE_RX     0x140
                MX8MP_IOMUXC_UART1_TXD_UART1_DCE_TX     0x140
        >;
};

字符串的第一部分 MX8MP_IOMUXC_UART1_RXD_UART1_DCE_RX 指定了引脚（在这个例子中是 UART1_RXD）。字符串的第二部分（UART1_DCE_RX）是该引脚所选的复用项。引脚设置值（右侧的十六进制值）定义了引脚的不同模式，例如，内部拉电阻是否被激活。在当前情况下，内部拉电阻被禁用。

UART1引脚复用的设备树表示：https://github.com/phytec/linux-phytec/blob/v6.6.21-phy1/arch/arm64/boot/dts/freescale/imx8mp-phyboard-pollux-rdk.dts#L387

7.2. RS232/RS485

phyCORE-i.MX 8M Plus 支持最多 4 个 UART 单元。在 phyBOARD-Pollux 上，UART1（调试串口）和 UART4 的 TTL 电平信号被路由到 Silicon Labs CP2105 UART 到 USB 转换IC。这一 USB 信号通过 Micro-USB 连接器 X1 输出。UART3 位于 X6（扩展连接器），为 TTL 电平信号。UART2 连接到一个多协议收发器IC，可用于 RS-232 和 RS-485，提供在 RS-232 电平的引脚连接器 X2 或 RS-485 电平的连接。多协议配置通过主板上的跳线 JP3 和 JP4 完成。有关正确设置的更多信息，请参阅 phyCORE-i.MX 8M Plus/phyBOARD-Pollux 硬件手册中的 UARTs 部分。

我们对RS-232和RS-485使用相同的设备树节点。RS485模式可以通过ioctl TIOCSRS485 来启用。此外，双向通讯支持也可以通过ioctl进行配置。请查看我们的小示例应用程序rs485test，该程序也包含在BSP中。跳线 JP3 和 JP4 需要正确设置。

7.2.1. RS232

    target:~$ stty -a

    target:~$ stty -F /dev/ttymxc1 115200 crtscts raw -echo

  • 通过简单的echo和cat，可以测试基本的通信。示例：

    target:~$ echo 123 > /dev/ttymxc1

    host:~$ cat /dev/ttyUSB2

主机应该打印出 “123”。

ethernet1 = eth0

7.4. SD/MMC 卡

该 i.MX 8M Plus 支持一个插槽，用于SD卡和MMC卡，作为linux通用块设备。这些设备可以像其他任何块设备一样使用。

警告

这些设备是热插拔的。然而，您必须确保在设备仍然挂载时不要拔掉它。这可能会导致数据丢失！

插入SD/MMC卡后，内核将在/dev中生成新的设备节点。完整设备可以通过其/dev/mmcblk1设备节点访问。SD/MMC卡的分区将显示为：

/dev/mmcblk1p<Y>

<Y> 作为分区编号，从1开始计数，直到该设备的最大分区数量。分区可以使用任何类型的文件系统进行格式化，并且可以以标准方式进行处理，例如，mount 和 umount 命令按预期工作。

小技巧

这些分区设备节点仅在SD卡包含有效的分区表时可用（类似于“硬盘”的处理）。如果没有分区表，则整个设备可以作为文件系统使用（类似于“软盘”的处理）。在这种情况下，必须使用 /dev/mmcblk1 进行格式化和挂载。卡片始终以可写方式挂载。

7.5. eMMC设备

PHYTEC模块如phyCORE-i.MX 8M Plus 配备了eMMC存储芯片作为主要存储。eMMC设备使用多层单元（MLC）或三层单元（TLC）技术来实现存储，并结合了处理ECC和磨损均衡的存储控制器。它们通过SD/MMC接口连接到 i.MX 8M Plus ，并在Linux内核中作为块设备表示，如SD卡、优盘或硬盘。

7.5.2. 启用后台操作 (BKOPS)

与原始NAND Flash相比，eMMC设备包含一个闪存传输层（FTL），该层负责处理原始MLC或TLC的磨损均衡、块管理和错误更正码（ECC）。这需要定期执行一些维护任务（例如擦除未使用的块）。这些任务被称为 后台操作（BKOPS） 。

默认情况下（取决于芯片），后台操作可能会定期执行，也可能不会，这会影响最坏情况下的读写延迟。

JEDEC标准自版本v4.41起规定了一种方法，主机可以手动发出BKOPS。有关更多详细信息，请参阅JEDEC标准章节“背景操作”以及eMMC数据手册中寄存器BKOPS_EN（寄存器：163）和BKOPS_START（寄存器：164）的描述。

寄存器 BKOPS_EN（寄存器：163）的位 MANUAL_EN（位 0）的含义：

  • 值 0：主机不支持手动触发 BKOPS。设备写入性能受到影响。

  • 值1：主机支持手动触发BKOPS。当主机不需要设备时，它会不时发出BKOPS。

自v3.7版本以来，Linux内核已经实现了发起后台操作的机制。您只需在eMMC设备上启用BKOPS_EN（详细信息见下文）。

JEDEC标准v5.1引入了一种新的自动BKOPS功能。它使主机能够定期触发后台操作，因为设备在空闲时会自动启动BKOPS（请参见寄存器BKOPS_EN（寄存器：163）中位AUTO_EN的描述）。

  • 要检查 BKOPS_EN 是否已设置，请执行：

    target:~$ mmc extcsd read /dev/mmcblk2 | grep BKOPS_EN

    输出将会是，例如：

    Enable background operations handshake [BKOPS_EN]: 0x01
#OR
Enable background operations handshake [BKOPS_EN]: 0x00

    值0x00表示BKOPS_EN被禁用，设备的写入性能受到影响。值0x01表示BKOPS_EN被启用，主机将不时发起后台操作。

  • 启用可以通过以下命令完成：

    target:~$ target:~$ mmc --help

[...]
mmc bkops_en <auto|manual> <device>
    Enable the eMMC BKOPS feature on <device>.
    The auto (AUTO_EN) setting is only supported on eMMC 5.0 or newer.
    Setting auto won't have any effect if manual is set.
    NOTE!  Setting manual (MANUAL_EN) is one-time programmable (unreversible) change.

  • 要设置BKOPS_EN位，请执行：

    target:~$ mmc bkops_en manual /dev/mmcblk2

  • 为了确保新设置生效并且内核能够自动触发BKOPS，请关闭系统：

    target:~$ poweroff

小技巧

BKOPS_EN位是一次性可编程的，无法恢复。

7.5.5. 启用伪SLC模式

eMMC设备使用 MLC或TLC 来存储数据。与NAND Flash中使用的SLC相比，MLC或TLC在成本更低的情况下，可靠性较低且错误率较高。

如果您更喜欢可靠性而不是存储容量，可以启用伪SLC模式或SLC模式。这里使用的方法采用了增强属性，该属性在JEDEC标准中有所描述，可以为设备的连续区域设置。JEDEC标准并未规定增强属性的实现细节和保障，这由芯片制造商自行决定。对于美光（Micron）芯片，增强属性提高了可靠性，但也将容量减半。

警告

在设备上启用增强属性时，所有数据将被丢失。

以下步骤展示了如何启用增强属性。

  • 首先使用以下命令获取eMMC设备的当前大小：

    target:~$ parted -m /dev/mmcblk2 unit B print

7.5.6. 擦除设备

可以直接擦除eMMC设备，而不是通过写零覆盖它。eMMC块管理算法将擦除底层的MLC或TLC，或者将这些块标记为可丢弃。设备上的数据将丢失，并将被读取为零。

  • 从SD卡启动后执行：

    target:~$ blkdiscard -f --secure /dev/mmcblk2

    选项 –secure 确保命令在 eMMC 设备擦除所有块之前会等待。-f (强制) 选项在擦除之前禁用所有检查，当 eMMC 设备包含已有数据的分区时需要使用此选项。

小技巧

target:~$ dd if=/dev/zero of=/dev/mmcblk2 conv=fsync

还会销毁设备上的所有信息，但这个命令对磨损均衡不好，并且需要更长的时间！

7.5.7. eMMC 启动分区

eMMC设备包含四个不同的硬件分区：User分区、boot1分区、boot2分区和rpmb分区。

User分区在JEDEC标准中称为用户数据区，是主要的存储分区。分区boot1和boot2可以用于存放bootloader，并且更可靠。 i.MX 8M Plus 使用哪个分区加载bootloader由eMMC设备的引导配置控制。分区rpmb是一个小分区，只能通过受信任的机制访问。

此外，User分区可以分为四个用户定义的一般用途区域分区。对此功能的解释超出了本文件的范围。有关更多信息，请参阅JEDEC标准第7.2章分区管理。

小技巧

不要将eMMC分区与DOS、MBR或GPT分区表的分区混淆。

当前的PHYTEC BSP没有使用eMMC设备的额外分区功能。U-Boot被烧写到用户分区的开始位置。U-Boot环境被放置在U-Boot之后的固定位置。使用MBR分区表创建两个分区，一个是FAT32引导分区，另一个是ext4根文件系统分区。它们位于U-Boot和U-Boot环境之后。FAT32引导分区包含内核和设备树。

7.6. SPI主设备

i.MX 8M Plus 控制器包含一个 FlexSPI 和一个 ECSPI IP 核。FlexSPI 主控制器支持两个 SPI 通道，最多可连接 4 个设备。每个通道支持单通道/双通道/四通道/八通道模式的数据传输（1/2/4/8 条双向数据线）。ECSPI 控制器支持 3 个 SPI 接口，每个接口都有一个专用的CS（chip select）引脚。由于CS也可通过 GPIO 实现，因此每个通道上可以连接多个设备。

在设备树中，SPI主节点的定义可以在这里找到：https://github.com/phytec/linux-phytec/blob/v6.6.21-phy1/arch/arm64/boot/dts/freescale/imx8mp-phycore-som.dtsi#L67

7.7. GPIOs

phyBOARD-Pollux 具有一组专门用于用户输入/输出的引脚。这些引脚直接连接到 i.MX 8M Plus 引脚，并被复用为 GPIO。它们可以直接在 Linux 用户空间中使用。处理器将其 GPIO 组织为五个每个包含 32 个 GPIO 的组（GPIO1 – GPIO5）。gpiochip0、gpiochip32、gpiochip64、gpiochip96 和 gpiochip128 是这些内部 i.MX 8M Plus GPIO 组 GPIO1 – GPIO5 的 sysfs 表示。

GPIO被标识为GPIO<X>_<Y>（例如：GPIO5_07）。<X>表示GPIO Bank，从1计数到5，而<Y>表示该Bank内的GPIO。<Y>从0计数到31（每个bank有32个GPIO）。

相比之下，Linux内核使用一个单一的整数来枚举系统中所有可用的GPIO。计算正确数字的公式是：

Linux GPIO number: <N> = (<X> - 1) * 32 + <Y>

从用户空间访问GPIO将使用libgpiod。它提供了一个库和工具，用于与Linux GPIO字符设备进行交互。以下是一些工具的用法示例：

gpiochip2 [30220000.gpio] (32 lines)

    target:~$ gpioget -c gpiochip0 20

7.10. EEPROM

在 phyCORE-i.MX8MP 上贴了一个 i2c 接口的 EEPROM 存储。它有两个地址。主 EEPROM 空间（总线：I2C-0 地址：0x51）可以通过 Linux 中的 sysfs 接口访问。主 EEPROM 的前 256 字节和 ID 页（总线：I2C-0 地址：0x59）用于板检测，不可修改。因此，ID 页不能通过 sysfs 接口访问。覆盖保留空间将导致启动问题。

如果您删除了保留的EEPROM空间，请联系我们的支持团队！

7.10.1. phyCORE-i.MX8MP 上的I2C EEPROM

警告

EEPROM ID页面（总线：I2C-0 地址：0x59）和正常EEPROM区域的前265个字节（总线：I2C-0 地址：0x51）不可被擦除或修改。这将影响bootloader的行为。板子可能无法正确启动。

phyCORE-i.MX8MP SoM上的I2C EEPROM连接到I2C-0总线的I2C地址0x51。可以直接对该设备进行读写操作：

target:~$ hexdump -c /sys/class/i2c-dev/i2c-0/device/0-0051/eeprom

要读取并以十六进制打印 EEPROM 的前 1024 字节，请执行：

target:~$ dd if=/sys/class/i2c-dev/i2c-0/device/0-0051/eeprom bs=1 count=1024  | od -x

要用零填充4KiB的EEPROM（总线：I2C-0 地址：0x51），并保留EEPROM数据，请使用：

target:~$ dd if=/dev/zero of=/sys/class/i2c-dev/i2c-0/device/0-0051/eeprom seek=1 bs=256 count=15

7.10.2. EEPROM SoM 检测

在|som| 上配置的I2C EEPROM具有一个可通过I2C地址0x59在i2c0上寻址的独立ID页面，以及一个可通过I2C地址0x51在i2c0上寻址的正常区域。PHYTEC使用这个32字节的数据区域来存储关于SoM的信息，包括PCB版本和配置。

在启动的早期阶段读取EEPROM数据。它用于选择正确的DDR RAM配置。这使得可以使用相同的bootloader镜像来支持不同的RAM大小，并自动选择正确的DTS overlay。

如果EEPROM ID页面数据和正常区域的前265个字节被删除，bootloader程序将回退到|som| Kit RAM设置，即 2GiB RAM。

7.11.1. RTC参数

实时时钟（RTC）具有一些功能，可以通过 hwclock 工具进行读取和设置。

  • 我们可以通过以下方式检查RTC支持的功能：

    target:~$ hwclock --param-get features
The RTC parameter 0x0 is set to 0x71.

    这个值的含义在内核中进行了编码，每个位被设置为1的位翻译为：

    #define RTC_FEATURE_ALARM               0
#define RTC_FEATURE_ALARM_RES_MINUTE    1
#define RTC_FEATURE_NEED_WEEK_DAY       2
#define RTC_FEATURE_ALARM_RES_2S        3

#define RTC_FEATURE_BACKUP_SWITCH_MODE  6
#define RTC_FEATURE_ALARM_WAKEUP_ONLY   7
#define RTC_FEATURE_CNT                 8

    target:~$ hwclock --param-get bsm

    target:~$ hwclock --param-set bsm=0x2

7.16. 电源管理

7.16.1. CPU核心频率调节

i.MX 8M Plus SoC中的CPU能够调整时钟频率和电压。这用于在不需要CPU的全部性能时节省电力。调整频率和电压被称为“动态电压和频率调整”（DVFS）。i.MX 8M Plus BSP支持DVFS功能。Linux内核提供了一个DVFS框架，允许每个CPU核心设置最小或最大频率和一个管理其运行的governor。根据使用的 i.MX 8 变体，支持几种不同的频率。

小技巧

尽管DVFS框架为每个CPU核心提供了频率设置，但一个CPU核心的频率设置会影响其他CPU核心。因此，所有CPU核心始终共享相同的DVFS设置。每个核心的单独DVFS设置是不可能的。

  • 要获取完整列表，请输入：

    target:~$ cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_available_frequencies

    例如 i.MX 8MPlus CPU，最高可达约 1.6 GHz，则结果将是：

    1200000 1600000

  • 要询问当前的频率类型：

    target:~$ cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_cur_freq

governor 会根据它们的目标自动选择这些频率中的一个。

  • 列出所有可用的 governor，使用以下命令：

    target:~$ cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_available_governors

    结果是：

    ondemand userspace performance schedutil

  • ondemand （默认）根据当前系统负载在可能的CPU核心频率之间切换。当系统负载超过特定限制时，它会立即提高CPU核心频率。

  • performance 始终选择最高的CPU核心频率。

  • userspace 允许以root身份运行的用户或用户空间程序设置特定频率（例如，设置为1600000）。输入：

  • 要查询当前的 governor，请输入：

    target:~$ cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_governor

    您通常会得到：

    schedutil

  • 切换到另一个governor（例如，userspace）可以通过以下方式完成：

    target:~$ echo userspace > /sys/devices/system/cpu/cpu0/cpufreq/scaling_governor

  • 现在你可以设置频率：

    target:~$ echo 1600000 > /sys/devices/system/cpu/cpu0/cpufreq/scaling_setspeed

有关调节器的更详细信息，请参阅Linux内核代码库中的Linux内核文档，路径为 Documentation/admin-guide/pm/cpufreq.rst 。

7.16.2. CPU核心管理

该 i.MX 8M Plus SoC 芯片上可以有多个处理器核心。例如，该 i.MX 8M Plus 具有 4 个 ARM 核心，可以在运行时单独开启和关闭。

  • 要查看系统中所有可用的核心，请执行：

    target:~$ ls /sys/devices/system/cpu  -1

  • 这将显示，例如：

    cpu0    cpu1   cpu2   cpu3   cpufreq
[...]

    这里系统有四个处理器核心。默认情况下，系统中所有可用的核心都被启用，以获得最佳性能。

  • 要关闭某个核，请执行：

    target:~$ echo 0 > /sys/devices/system/cpu/cpu3/online

    作为确认，您将看到：

    [  110.505012] psci: CPU3 killed

    现在核心已关闭电源，并且该核心上不再调度任何进程。

  • 您可以使用 top 命令查看核心和进程的图形概览：

    target:~$ htop

  • 要重新启动核心，请执行：

    target:~$ echo 1 > /sys/devices/system/cpu/cpu3/online

7.17. 热管理

7.17.1. U-Boot

之前U-Boot中的温度控制不够理想。现在，U-Boot增加了温度关机功能，以防止在启动过程中板子过热。关机发生的温度与内核中的温度一致。

当前温度的各个温度范围在启动日志中显示：

CPU:   Industrial temperature grade (-40C to 105C) at 33C

7.17.2. 内核

Linux内核集成了热管理功能，能够监测系统芯片（SoC）温度，降低CPU频率，控制风扇，通知其他驱动程序减少功耗，并在最坏的情况下优雅地关闭系统（https://www.kernel.org/doc/Documentation/thermal/sysfs-api.txt）。

本节描述了如何在 i.MX 8M Plus SoC 平台上使用热管理内核 API。 i.MX 8 具有用于 SoC 的内部温度传感器。

  • 当前温度可以以毫摄氏度为单位读取：

    target:~$ cat /sys/class/thermal/thermal_zone0/temp

  • 例如，你将得到：

    49000

imx_thermal内核驱动注册了两个触发点。这些触发点根据CPU变体的不同而有所不同。区分工业版和商业版。

（请查看内核 sysfs 文件夹 /sys/class/thermal/thermal_zone0/）

内核热管理使用这些触发点来触发事件并实施冷却措施。内核中可用的热政策（也称为thermal governor）包括：Step Wise、Fair Share、Bang Bang和Userspace。BSP中使用的默认策略是Step Wise。如果sysfs文件temp中的SoC温度值高于 trip_point_0 ，则CPU频率将设置为最低CPU频率。当SoC温度再次降到 trip_point_0 以下时，限制将被解除。

7.18. 看门狗

PHYTEC i.MX 8M Plus 模块包含一个硬件看门狗，当系统挂起时能够重置板子。看门狗在 U-Boot 中默认启动，超时时间为 60 秒。因此，即使在早期内核启动过程中，看门狗也已经开始运行。Linux 内核驱动程序控制看门狗，并确保它有被踢到。本节将解释如何使用 systemd 在 Linux 中配置看门狗，以检查系统挂起和重启期间的情况。

7.18.1. systemd中的看门狗支持

Systemd 从版本 183 开始支持硬件看门狗。

  • 要启用看门狗支持，需要通过启用选项来配置 /etc/systemd/ 中的文件system.conf文件：

    RuntimeWatchdogSec=60s
ShutdownWatchdogSec=10min

RuntimeWatchdogSec 定义了看门狗的超时时间，而 ShutdownWatchdogSec 定义了系统重启时的超时时间。有关 systemd 下硬件看门狗的更多详细信息，请访问 http://0pointer.de/blog/projects/watchdog.html。更改将在重启后生效，或者运行：

target:~$  systemctl daemon-reload

7.19. snvs电源键

连接到开/关按钮的 X_ONOFF 引脚可以长按以触发关机，而无需软件干预。使用 snvs_pwrkey 驱动程序时，当按下按钮时，KEY_POWER 事件也会报告给用户空间。默认情况下，systemd 被配置为忽略此类事件。无软件干预的关机功能没有配置。可以在 /etc/systemd/logind.conf 中配置在按下开/关按钮时通过 systemd 触发关机，并使用以下设置：

HandlePowerKey=poweroff
